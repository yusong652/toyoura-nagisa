# Shell Command & PFC Python Console Implementation Plan

**Branch**: `feature/shell-and-pfc-console`
**Date**: 2025-12-13

## Overview

Implement Claude Code-style shell command injection and PFC Python console for enhanced human-AI collaboration in the toyoura-nagisa CLI.

### Inspiration: Claude Code's `!` Command Design

Claude Code injects user's shell command results into the conversation context with a clever pattern:

```
Caveat: The messages below were generated by the user while running local commands.
DO NOT respond to these messages or otherwise consider them in your response
unless the user explicitly asks you to.

<bash-input>date</bash-input>
<bash-stdout>Sat Dec 13 03:02:39 JST 2025</bash-stdout>
<bash-stderr></bash-stderr>
```

**Key Design Principles**:
- LLM can "see" user operations (context awareness)
- Default behavior: don't respond (reduce noise)
- User can explicitly ask about the output (flexibility)

## Command Prefixes

| Prefix | Target | Example |
|--------|--------|---------|
| `!` | Local shell | `!git status`, `!ls` |
| `>` | PFC Python | `> itasca.command("ball create id 1 ...")` |

## Key Design Decisions

### 1. Direct Python API Mode for PFC

**Decision**: User inputs Python code directly, exactly like in PFC GUI's IPython console.

```python
# User inputs Python code directly (no automatic wrapping)
> itasca.command("ball create id 1 x 0 y 0 z 0 radius 0.5")
> ball = itasca.ball.find(1)
> print(ball.pos())
> for i in range(10): itasca.command(f"ball create id {i+10} x {i} y 0 z 0 radius 0.5")
```

**Rationale**:
- Identical experience to PFC GUI IPython - no learning curve
- Full Python flexibility (variables, loops, conditionals)
- User controls exactly what code runs
- Consistency with `pfc_execute_task` script mode (both are Python)
- Natural progression from quick commands to full scripts

### 2. Task History Integration

**Decision**: Quick commands SHOULD be recorded in task history.

**Rationale**: PFC project is stateful. Every command changes simulation state.

```
Without history:
  tasks: []
  Actual PFC state: State C (LLM doesn't know how it got there)

With history:
  tasks: [quick#1, quick#2, quick#3]
  LLM can trace: State A → B → C
```

### 3. Task Type Differentiation

| Field | Quick Command | Script Task |
|-------|---------------|-------------|
| `type` | `"quick"` | `"script"` |
| `source` | `"user_shell"` | `"agent"` / `"user"` |
| `git_snapshot` | `false` | `true` (if background) |
| `script_content` | Single-line wrapped | Full script |
| `run_in_background` | `false` | Configurable |

### 4. Display in pfc_list_tasks

```
Tasks:
  #1 [quick]  ball create id 1...        ✓ success  (user_shell)
  #2 [quick]  ball create id 2...        ✓ success  (user_shell)
  #3 [script] compression_test.py        ⏳ running (agent)
```

## Architecture

### CLI Components (packages/cli)

```
InputParser
├── detectCommandPrefix(input: string)
│   ├── '!' → { type: 'shell', command: string }
│   ├── '>' → { type: 'pfc_python', code: string }
│   └── null → regular chat message
│
├── LocalShellExecutor (STATEFUL - persistent shell session)
│   ├── spawn() → Initialize persistent shell process
│   ├── execute(command) → { stdout, stderr, exitCode }
│   ├── getCwd() → Current working directory
│   └── dispose() → Cleanup shell process
│
└── PFCPythonExecutor
    └── execute(code) → { output, returnValue, error, taskId }

ContextInjector
└── formatForLLM(type, input, output) → Claude Code compatible format
    ├── shell → <bash-input>/<bash-stdout>/<bash-stderr>
    └── pfc   → <pfc-python>/<input>/<output>/<error>
```

### Backend Components (packages/backend)

```
pfc-server modifications:
├── New message type: "quick_python"
│   └── { type: "quick_python", code: string }
│
├── TaskManager extensions:
│   ├── registerQuickPython(code) → taskId
│   └── Task model: add `task_type: "quick" | "script"`
│
└── Response format:
    └── { task_id, output, return_value, error, task_type }
```

### WebSocket Protocol Extension

```jsonc
// Request (CLI → pfc-server)
{
  "type": "quick_python",
  "code": "itasca.command(\"ball create id 1 x 0 y 0 z 0 radius 0.5\")"
}

// Response (pfc-server → CLI)
{
  "type": "quick_python_result",
  "task_id": "quick_001",
  "output": "Ball created: id=1",
  "return_value": null,  // Python return value if any
  "error": null
}
```

## Architecture: User Terminal vs Agent Bash Tool

A key insight from Claude Code: **User terminal is stateful, Agent bash tool is stateless**.

| Feature | User Terminal (`!`) | Agent Bash Tool |
|---------|---------------------|-----------------|
| State | **Stateful** (preserves cwd/env) | **Stateless** (each invocation independent) |
| Format | `bash-input/stdout/stderr` | Same format (reused logic) |
| Session | Persistent shell session | Fresh execution context |
| Purpose | Human operation context awareness | Agent automated execution |

**Implementation Implication**:
- User terminal needs a persistent shell session (e.g., `child_process.spawn` with persistent stdin/stdout)
- Agent bash tool can use simple `exec()` per command
- Both use the same output format for LLM consumption

## Context Injection Format

**Important**: Match Claude Code's exact format for consistency and potential future compatibility.

### Shell Command (`!`)

```
<bash-input>git status</bash-input>
<bash-stdout>On branch feature/shell-and-pfc-console
nothing to commit, working tree clean</bash-stdout>
<bash-stderr></bash-stderr>
```

### PFC Python Command (`>`)

```xml
<pfc-python>
<input>itasca.command("ball create id 1 x 0 y 0 z 0 radius 0.5")</input>
<task_id>quick_001</task_id>
<output>Ball created: id=1</output>
<error></error>
</pfc-python>
```

## Open Questions

1. **Prefix choice for PFC**: `>` vs `pfc>` vs `py>`?
   - `>` is concise but might conflict with shell redirect
   - `pfc>` is explicit but verbose
   - **Current choice**: `>` (revisit if issues arise)

2. **Multi-line PFC commands**: How to handle?
   - Option A: Not supported in quick mode (use script for complex commands)
   - Option B: Use `>>` to enter multi-line mode
   - **Current choice**: Option A (keep quick mode simple)

3. **Command history**: Should CLI maintain separate history for shell/PFC?
   - Could use different arrow-key behaviors
   - **Decision**: Defer to Phase 3
